# Scheduling & Capacity Pitfalls

- [Pitfall] `WORKER_MAX_WEIGHT_OVERRIDE` above physical RAM is dangerous — Override above available GB silently logs `tracing::error!("risk of OOM")` but still applies. Trigger: env-tuning by an operator who doesn't understand the topology. Correct approach: only override when intentionally allowing oversubscription; monitor logs. Source: `crates/worker/src/limiter.rs get_max_weight`. Affected module: `crates/worker`.
- [Pitfall] `estimate_duration` is a heuristic — Returns fixed ms (Controller=200, PlonkWrap=8000, ProveShard=2000, …). Real durations vary 10×+. Assignment by lowest `next_free_time` can pile work on a single worker if estimates are wrong. Trigger: assuming `next_free_time` is accurate. Correct approach: rely on heartbeat-driven authoritative `current_weight` for true capacity. Source: `bin/coordinator/src/lib.rs`. Affected module: `bin/coordinator`.
- [Pitfall] Worker weight is updated authoritatively from heartbeat — The coordinator's incremental weight bookkeeping in `assign_task` / `complete_task` / `fail_task` can drift, but each heartbeat clobbers it. Trigger: trusting incremental weight between heartbeats. Correct approach: tolerate brief incorrect capacity decisions between 5s heartbeats. Source: `bin/coordinator/src/lib.rs handle_heartbeat`. Affected module: `bin/coordinator`.
- [Pitfall] Balanced policy holds queue-and-weight invariants only inside `state` write lock — `proof_gpu_weights` is decremented in `post_task_update_state` and cleared in `post_worker_empty`/`on_proof_deleted`. A coordinator crash between `worker.weight -= w` and `P::post_task_update_state` can leave inflated GPU weight and de-prioritize the proof. Trigger: panic between the two steps. Correct approach: keep both updates inside the same critical section. Source: `bin/coordinator/src/policy/balanced.rs`. Affected module: `bin/coordinator`.
- [Pitfall] Coordinator state RW lock — never reacquire while holding subscriber — `add_subscriptions` drops the sub DashMap entry before acquiring `state.write_owned()`; otherwise deadlock. Trigger: refactor to hold both. Correct approach: always `drop(sub)` before acquiring state. Source: comment "Sub cannot be acquired while acquiring state" in `bin/coordinator/src/lib.rs`. Affected module: `bin/coordinator`.
- [Pitfall] Task channels persist 60s after close — `STALE_THRESHOLD` lets late subscribers replay buffered messages. Trigger: manual cleanup racing the cleanup loop. Correct approach: let `cleanup_stale_task_channels` reap channels naturally. Source: `bin/coordinator/src/lib.rs`. Affected module: `bin/coordinator`.
- [Pitfall] `MarkerDeferredRecord` warnings are suppressed — "worker was not working on task" log is downgraded for marker tasks because they're synthetic. Trigger: adding new marker-like task types. Correct approach: replicate the suppression. Source: `bin/coordinator/src/lib.rs complete_task` else-branch. Affected module: `bin/coordinator`.
- [Pitfall] `spawn_proof_claimer_task` polls every 500ms — Hot loop against the cluster API; lowering the bound increases load, raising changes E2E latency. Trigger: chasing latency or load. Correct approach: tune deliberately and measure end-to-end. Source: `bin/coordinator/src/cluster.rs`. Affected module: `bin/coordinator`.
- [Pitfall] Worker drain bound is 30 minutes — Tasks still running at `DRAIN_TIMEOUT` are killed mid-execution and rely on heartbeat timeout (30s) to be reassigned. Trigger: deploying with ECS `stopTimeout` < 1h. Correct approach: stay under ECS `stopTimeout=3600s`. Source: `bin/node/src/main.rs`. Affected module: `bin/node`.
- [Pitfall] Task hard timeout = 6h — Any task running > `TASK_TIMEOUT` is force-aborted to prevent deadlock-bug accumulation. Trigger: legitimate long-running task. Correct approach: when adding long-running tasks ensure they fit; do not bypass. Source: `bin/node/src/main.rs`. Affected module: `bin/node`.
- [Pitfall] ctrl+c handler must CAS the shutdown flag — Two near-simultaneous signals could both enter the first branch and skip the force-exit path. Trigger: rapid double ctrl+c. Correct approach: use `compare_exchange(false, true, AcqRel, Acquire)` to gate the first-signal branch. Source: `was_first_signal` in `bin/node/src/main.rs`. Affected module: `bin/node`.
- [Pitfall] Bidder aggressive mode bypasses capacity/time checks — When `aggressive_mode == true`, only optional `min_deadline_secs` applies. Trigger: enabling aggressive mode in untrusted environments. Correct approach: reserve for permissioned environments. Source: `bin/bidder/src/lib.rs bid_requests`. Affected module: `bin/bidder`.
- [Pitfall] Fulfiller `time_now` panics on system clock skew — "time went backwards" expect crashes the fulfiller. Trigger: backward NTP jumps. Correct approach: run with monotonic-aware NTP. Source: `crates/fulfillment/src/lib.rs time_now`. Affected module: `crates/fulfillment`.
