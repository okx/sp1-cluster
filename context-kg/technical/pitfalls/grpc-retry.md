# gRPC Retry & Reconnect Pitfalls

- [Pitfall] Heartbeat retries are infinite — `WorkerServiceClient::heartbeat / complete_task / fail_task / ack_sub` use `retry::infinite()`; they block forever if coordinator is down. Trigger: assuming a bounded retry. Correct approach: reserve `retry::bounded(Duration)` for calls whose `Err` is a caller-acted signal. Source: `retry` module comments in `crates/worker/src/client.rs`. Affected module: `crates/worker`.
- [Pitfall] Each call site MUST pick `retry::infinite()` or `retry::bounded(Duration)` explicitly — There is no shared default; failure semantics cannot be inherited silently. Trigger: introducing a helper that hides the retry policy. Correct approach: keep the choice visible per call site. Source: `crates/worker/src/client.rs`. Affected module: `crates/worker`.
- [Pitfall] Subscriber `update_sub` sends max 30 task ids — Hard-coded `choose_multiple(rng, 30)` random subset on each heartbeat. Trigger: raising the cap without coordinator-side rate-limiting. Correct approach: pair any cap change with rate limiting on the server side. Source: `ServerHeartbeat` branch in `crates/worker/src/client.rs`. Affected module: `crates/worker`.
- [Pitfall] Worker heartbeat reconnect triggers specifically on `tonic::Code::NotFound` — Coordinator must return `NotFound` (not `InvalidArgument`) when a worker re-heartbeats after being purged by `cleanup_dead_workers`; the code is the only signal. Trigger: changing the error code in coordinator. Correct approach: keep `NotFound` for "worker not registered" responses; document the contract. Source: `bin/node/src/main.rs` worker loop. Affected module: `bin/node`.
- [Pitfall] tonic open_sub is not cancel-safe — `open_sub` and `ack_sub` are flagged not cancel-safe; wrapping them in `tokio::time::timeout` or `tokio::select!` can corrupt subscriber state. Trigger: adding timeout wrappers. Correct approach: never cancel a pending `open_sub`/`ack_sub` mid-flight. Source: comments in `bin/coordinator/src/server.rs`. Affected module: `bin/coordinator`, `crates/worker`.
- [Pitfall] `reconnect_with_backoff` must install rustls ring provider — TLS gRPC requires the provider; helper is idempotent but explicit installs in `main.rs` of each binary still need it. Trigger: skipping `rustls::crypto::ring::default_provider().install_default()` in a new binary's main. Correct approach: always install ring as the first action after `dotenv()`. Source: `crates/common/src/client.rs` and per-binary `main.rs`. Affected modules: all binaries.
- [Pitfall] `configure_endpoint` constants must stay in lockstep across binaries — `crates/fulfillment/src/grpc.rs` and `bin/bidder/src/grpc.rs` declare identical timeout/keepalive constants (10s/5s/10s/10s/30s). Trigger: tuning one without the other. Correct approach: change both copies together. Source: both `grpc.rs` files. Affected modules: `crates/fulfillment`, `bin/bidder`.
- [Pitfall] tonic graceful shutdown 2s sleep can cut long-running RPCs — Coordinator gives 2s sleep post-shutdown then signals server stop. Trigger: long-running RPCs that exceed 2s. Correct approach: keep RPCs short or accept the cut; do not add a sleep loop that violates ECS stopTimeout. Source: `bin/coordinator/src/server.rs`. Affected module: `bin/coordinator`.
- [Pitfall] Bounded 10s retry on `ClusterServiceClient` may hide upstream outages — Raising it can mask outages; lowering it can cause spurious caller errors. Trigger: tuning to chase symptoms rather than the cause. Correct approach: investigate root cause before re-tuning. Source: `crates/common/src/client.rs`. Affected module: `crates/common`.
