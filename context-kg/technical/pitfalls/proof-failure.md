# Proof Failure Pitfalls

- [Pitfall] `TaskError::Execution` reports `TaskStatus::Succeeded` but unclaims the proof — `run_task` returns `Succeeded` to coordinator AND calls `try_unclaim_proof` which writes `Failed`. Two paths race to update the DB. Trigger: any executor-side error inside a CoreExecute/ExecuteOnly handler. Correct approach: do NOT change to `FailedFatal` — coordinator must not retry executions. Document the race; rely on `complete_proof(Failed)` arriving via `WorkerServiceClient`. Source: `crates/worker/src/lib.rs` `run_task` match. Affected module: `crates/worker`.
- [Pitfall] `try_unclaim_proof` runs for both Fatal AND Execution before coordinator's `enable_proof_fail` gate — Non-`enable_proof_fail` task types (ProveShard, RecursionDeferred, etc.) with Fatal/Execution errors still drag the whole proof to Failed. Trigger: introducing a new non-controller task type that fatals. Correct approach: be aware that worker-side unclaim is unconditional; coordinator's gate only governs the manual_proof_fail path. Source: `crates/worker/src/lib.rs`. Affected module: `crates/worker`.
- [Pitfall] `MAX_TASK_RETRIES = 3` increments only on retryable — Non-retryable failure → immediate FailedFatal regardless of retry count. Trigger: assuming retries apply to non-retryable errors. Correct approach: choose `TaskError::Retryable` vs `Fatal` deliberately at the handler boundary. Source: `bin/coordinator/src/lib.rs`. Affected module: `bin/coordinator`.
- [Pitfall] Heartbeat-driven reassignment doesn't bump retry count — A worker dying after 30s of silence (`WORKER_HEARTBEAT_TIMEOUT`) → its tasks re-enqueue without `task.retries += 1`. Flaky workers can replay the same task indefinitely. Trigger: deploying flaky workers (e.g. spotty network). Correct approach: monitor heartbeat reconnect rate; investigate workers that keep returning. Source: `bin/coordinator/src/lib.rs cleanup_dead_workers`. Affected module: `bin/coordinator`, `bin/node`.
- [Pitfall] `TASK_TIMEOUT = 6h` calls `fail_task(retryable=true)` — A truly stuck task only fails the proof after `MAX_TASK_RETRIES=3` retries — up to ~24h before a proof can fail this way (3 × 6h, ignoring re-assignment overhead). Trigger: long-running tasks that hang. Correct approach: if a task is known to be non-recoverable, use `TaskError::Fatal` instead of waiting for the timeout. Source: `bin/node/src/main.rs`. Affected module: `bin/node`.
- [Pitfall] DRAIN_TIMEOUT path emits no `fail_task` — At 30 min, the worker loop just breaks; coordinator finds out via 30s heartbeat timeout. ECS `stopTimeout=3600s` is the backstop. Trigger: long shutdowns. Correct approach: keep tasks short OR accept that abandoned tasks will retry once coordinator notices. Source: `bin/node/src/main.rs`. Affected module: `bin/node`.
- [Pitfall] `tasks.contains_key` early-return is the only dedupe for retransmits — Coordinator may retransmit `NewTask` after heartbeat reconciliation (`handle_heartbeat` re-sends task when worker says it doesn't have it). If the worker has already finished and removed the key, a late retransmit could re-run the task. Trigger: coordinator handle_heartbeat sending a stale `NewTask`. Correct approach: rely on the fact that re-running a task with the same ID is rare (the handler's removed entry would have to be repopulated); document the corner case. Source: `bin/node/src/main.rs` NewTask handler. Affected module: `bin/node`.
- [Pitfall] `complete_task` on already-Succeeded task is partial-no-op — Subscriber notify and `active_tasks` decrement are short-circuited, but `worker.weight` and `active_tasks` are still removed. Trigger: double-reporting on reconnect. Correct approach: the short-circuit is intentional to avoid double-decrementing `proof.active_tasks`. Source: `bin/coordinator/src/lib.rs complete_task`. Affected module: `bin/coordinator`.
- [Pitfall] `fail_proof_internal` re-inserts proof on validation failure — If a `task_id` was given but doesn't exist in the proof, the proof is re-inserted back into state before returning NotFound. Trigger: cross-instance fail_proof racing. Correct approach: maintain the rollback when refactoring. Source: `bin/coordinator/src/lib.rs`. Affected module: `bin/coordinator`.
- [Pitfall] `enable_proof_fail` task set must include all "no continuation" types — Currently Controller, Plonk/Groth16Wrap, ExecuteOnly, CoreExecute. Failing one of these manually fails the whole proof. Trigger: adding a new top-level task type. Correct approach: decide explicitly whether to include the new task type in `enable_proof_fail`. Source: `bin/coordinator/src/lib.rs`. Affected module: `bin/coordinator`.
- [Pitfall] Subscriber resend cap = 3 retries — Messages unacked for 2s are resent; after 3 resends the message is dropped. Trigger: assuming guaranteed delivery for subscribers. Correct approach: do not interpret message drop as success; bound subscriber resend logic before raising the cap. Source: `bin/coordinator/src/server.rs spawn_subscriber_channel_task`. Affected module: `bin/coordinator`.
- [Pitfall] `BalancedPolicy` panics on missing proof — `get_queued_task` panics with `proof not found` if the proof has been removed mid-queue. Trigger: race during shutdown. Correct approach: always queue tasks before removing proofs. Source: `bin/coordinator/src/policy/balanced.rs`. Affected module: `bin/coordinator`.
