# Redis Admission & Lifecycle Pitfalls

- [Pitfall] `AdmissionGuard` must outlive the upload — Binding the guard to `_` drops it immediately and reintroduces a TOCTOU race in admission accounting. The type is `#[must_use]`. Trigger: refactor that simplifies a `_guard = check_admission(...)` to `_ = check_admission(...)`. Correct approach: bind to a named `_guard` that lives until the write lands. Source: `crates/artifact/src/redis.rs` (`upload_to_transport`). Affected module: `crates/artifact`.
- [Pitfall] Redis `maxmemory` must be set — If `maxmemory` is unset (0), the lazy-init permit pool floors at `MIN_PERMITS_PER_NODE = 4` and caps cluster throughput at ~4 concurrent shards regardless of GPU count. Trigger: deploying with default Redis configuration. Correct approach: call `RedisArtifactClient::validate_config()` at startup (node + fulfiller both do); set `CONFIG SET maxmemory <N>gb` below the host/cgroup limit. Source: comments on `validate_config` and `compute_permits_for_node`. Affected module: `crates/artifact`.
- [Pitfall] Admission permit pool must remain below the admission ceiling — Compile-time `const _: () = assert!(INPUT_BUDGET_FRACTION < MEMORY_BUDGET_FRACTION)` (0.50 < 0.80) is load-bearing: closing the gap deadlocks the pipeline because non-permit-gated writes need headroom. Trigger: raising `INPUT_BUDGET_FRACTION`. Correct approach: never raise `INPUT_BUDGET_FRACTION` to or above `MEMORY_BUDGET_FRACTION`. Source: `crates/artifact/src/redis.rs`. Affected module: `crates/artifact`.
- [Pitfall] `HSETEX` unavailable on managed Redis — Multi-chunk uploads must use `HSET` + `EXPIRE` on the hash key. Trigger: refactoring to `HSETEX` for atomicity. Correct approach: keep `HSET` + `EXPIRE` (one artifact == one hash, whole-hash TTL equals per-field TTL). Source: comment above chunked upload in `par_upload_file`. Affected module: `crates/artifact`.
- [Pitfall] `ArtifactType::Program` must never have a TTL — Programs are deterministic-keyed and must persist; setting `EX`/`EXPIRE` on Program hash breaks the warm-cache contract used by network-gateway. Trigger: simplifying the upload to always EXPIRE. Correct approach: always check `!matches!(artifact_type, ArtifactType::Program)` before `EX`/`EXPIRE`. Source: `par_upload_file` Program branches. Affected module: `crates/artifact`, `bin/network-gateway`.
- [Pitfall] zstd level differs by transport — Redis uses level 0 (ephemeral, 4h TTL); S3 uses level 3 (persistent). Trigger: refactoring to unify compression. Correct approach: do not unify levels; comments call out the tradeoff. Source: comments in `upload_raw` of `redis.rs` and `s3.rs`. Affected module: `crates/artifact`.
- [Pitfall] `CompressedUpload::upload_raw_compressed` skips compression — Caller must have already zstd-compressed; double-compressing wastes CPU and breaks SDK symmetric pipeline. Trigger: invoking on raw payloads. Correct approach: only invoke from network-bucket copy paths where bytes are already zstd. Source: comments in both `redis.rs` and `s3.rs` `upload_raw_compressed` impls. Affected module: `crates/artifact`.
- [Pitfall] S3 download thread panic surfaces via `panic!` — An inner JoinHandle error or download error panics the entire process. Trigger: any per-chunk download failure. Correct approach: rely on AWS SDK's internal retry policy; the explicit `panic!("artifact download thread panicked: ...")` is non-recoverable by design. Source: `par_download_file` `set.join_next` match. Affected module: `crates/artifact`.
- [Pitfall] Refcount set `refs:{id}` must shard by artifact id — Sharding by a derived label would diverge refs from their artifact across nodes. Trigger: refactoring `get_redis_connection(id)` to use a different key shape. Correct approach: always use the artifact id as sharding key. Source: `crates/artifact/src/redis.rs`. Affected module: `crates/artifact`.
- [Pitfall] `ShardPermit::noop()` on closed semaphore is fail-open — Semaphore poisoning silently disables OOM protection. Trigger: process-wide semaphore close. Correct approach: monitor the integration test `scenario_3_no_oom_under_pressure_with_gate`; if it starts OOMing, the permit subsystem regressed. Source: `crates/artifact/src/redis.rs`. Affected module: `crates/artifact`.
- [Pitfall] Redis download `max_elapsed_time` = 1 s — At most ~1 second of retry budget across attempts despite a 60 s per-attempt timeout. Trigger: increasing concurrency without raising the budget. Correct approach: raise `with_max_elapsed_time` alongside any retry expansion. Source: `RedisArtifactClient::new`. Affected module: `crates/artifact`.

## Warnings

- [Warning] Redis admission fails open on `INFO memory` errors. A sick Redis (auth failure, network blip on the `INFO` call) silently disables backpressure rather than refusing writes. Monitor warn log "admission INFO memory failed; fail-open" — repeated occurrences mean memory protection is degraded.
